# 评估：基础规则内存设计（v2.1.7-dev）

> 版本：v2.1.7-dev
> 时间戳：2026-03-24
>
> 本文评估 `icu.windea.pls.config.config` 包中基础规则（`CwtPropertyConfig`、`CwtValueConfig`、`CwtOptionConfig`、`CwtOptionValueConfig`）的当前设计方案，并考量进一步的内存优化空间。

## 一、评估范围与背景

CWT 规则（`CwtConfig`）体系是整个插件的语义驱动核心——代码补全、解析、校验、文档等几乎所有语言功能都依赖对规则对象的高频访问。在支持 Stellaris / VIC3 / EU5 等大型游戏目录时，规则对象的总数量庞大（数以万计的 `CwtPropertyConfig` 实例），因此其内存布局和分配策略对插件的堆内存占用有直接影响。

**核心文件**：

- `CwtConfig.kt` — 统一抽象接口
- `CwtMemberConfig.kt` — 成员规则接口（sealed）
- `CwtPropertyConfig.kt` — 属性规则接口与实现
- `CwtValueConfig.kt` — 值规则接口与实现
- `CwtOptionConfig.kt` — 选项规则接口与实现
- `CwtOptionValueConfig.kt` — 选项值规则接口与实现
- `CwtOptionDataHolderBase.kt` — 选项数据持有器基类
- `CwtOptionDataHolder.kt` — 选项数据持有器接口（定义所有已知选项字段）
- `_accessors.kt` — 扩展属性与 UserData key 注册

**工具/基础设施**：

- `icu.windea.pls.core.optimizer.Optimizer` / `OptimizerRegistry` / `Optimizers.kt` — 优化器机制
- `icu.windea.pls.model._optimizers.kt` — `CwtType`、`CwtSeparatorType` 等枚举的优化器注册
- `icu.windea.pls.cwt.psi.CwtPropertyPointer` — 属性元素的复合智能指针
- `icu.windea.pls.inject.injectors.addon.InlinedDelegateFieldCodeInjectors` — 委托字段内联注入

## 二、类型层次结构

```
CwtConfig<T>
├── CwtMemberContainerConfig         // 持有子规则列表的规则
│   ├── CwtFileConfig                // 文件级规则
│   └── CwtMemberConfig (sealed)     // 成员规则
│       ├── CwtPropertyConfig        // 属性型（k = v 或 k = {...}）
│       └── CwtValueConfig           // 值型（v 或 {...}）
├── CwtDetachedConfig                // 脱离 PSI 的规则（无 configGroup / userdata）
│   └── CwtOptionMemberConfig (sealed) // 选项成员规则
│       ├── CwtOptionConfig          // 选项（## k = v）
│       └── CwtOptionValueConfig     // 选项值（## v）
└── CwtDelegatedConfig               // 委托型规则（包装另一规则，覆盖少量字段）
```

### 实现变体总览（以 `CwtPropertyConfig` 为例）

每种主要规则类型内部均有多个私有实现类，按"特征集合"分拆，遵循最小字段原则：

| 类名 | 用途 | 注释中的估算大小 |
|---|---|---|
| `CwtPropertyConfigImpl` | 叶节点（无子规则） | 48 bytes |
| `CwtPropertyConfigImplWithConfigs` | 块节点（有子规则） | 48 bytes |
| `CwtPropertyConfigDelegate` | 委托包装（无覆盖） | 32 bytes |
| `CwtPropertyConfigDelegateWithConfigs` | 委托包装（覆盖子规则） | 40 bytes |
| `CwtPropertyConfigDelegateWithKeyAndValue` | 委托包装（覆盖 key/value） | 40 bytes |

`CwtValueConfig` 的变体结构与之类似，并额外有 `CwtValueConfigFromPropertyConfig`（从属性规则推导，不重复存储字段）和 `CwtValueConfigMock`（模拟值规则，使用空指针）。

`CwtOptionConfig` / `CwtOptionValueConfig` 由于继承自 `CwtDetachedConfig`（不持有 PSI 指针和 configGroup），对象更轻量：`CwtOptionConfigImpl` 估算 24 bytes，`CwtOptionValueConfigImpl` 估算 16~24 bytes。

## 三、已有内存优化技术

这套实现已经落地了相当全面的内存优化手段，以下逐项说明。

### 3.1 枚举类型 → Byte 存储

**位置**：`CwtPropertyConfig.kt:266,281`、`CwtValueConfig.kt:224`、`CwtOptionConfig.kt:142,155`、`CwtOptionValueConfig.kt:91`

通过 `Optimizer<Enum, Byte>` 机制（见 `_optimizers.kt:12-30`），将 `CwtType` 和 `CwtSeparatorType` 枚举在实例字段中存储为 `Byte`（1 字节），访问时通过 `deoptimize` 反查 `entries[ordinal]`：

```kotlin
private val valueTypeId = valueType.optimized(OptimizerRegistry.forCwtType())
override val valueType: CwtType get() = valueTypeId.deoptimized(OptimizerRegistry.forCwtType())
```

每个字段从 4 字节对象引用（指向枚举单例）节省为 1 字节 Byte 字段。JVM 对象布局中 Byte 字段的实际内存节省取决于字段排列和填充，但在多个 Byte 字段连续时（如同时存储 `separatorTypeId` 和 `valueTypeId`）效果较好。

### 3.2 String interning

**位置**：`Optimizers.kt:39-46`（`StringOptimizer`）、`CwtOptionConfig.kt:144`、`CwtOptionValueConfig.kt:93`

对 `key`、`value` 等字符串字段通过 Caffeine `Interner.newWeakInterner()` 做弱引用 interning，相同内容的字符串在 JVM 内只保留一个实例。在大型游戏目录中，CWT 规则文件里存在大量重复的键名（如 `"type"`、`"scope"`、`"cardinality"` 等），interning 能显著减少 `String` 对象数量。

### 3.3 `configs` 列表 → ImmutableList

**位置**：`CwtPropertyConfig.kt:178,315`、`CwtValueConfig.kt:143,257`、`CwtFileConfig.kt:79`（均调用 `optimized()`）

`configs` 子规则列表在 `postProcess/postOptimize` 阶段通过 `ListOptimizer`（`Optimizers.kt:68-91`）转换为 Guava `ImmutableList`：

- 空列表 → `emptyList()`（单例）
- 单元素列表 → `ImmutableList.of(element)`（无多余容量）
- 多元素列表 → `ImmutableList.copyOf(input)`

相比 `ArrayList`（会预留额外容量、持有可变状态），`ImmutableList` 内存更紧凑，且有助于 JIT 做不变性分析。

### 3.4 CwtMembersType 延迟计算

**位置**：`CwtPropertyConfig.kt:301-310`、`CwtValueConfig.kt:243-252`、`CwtFileConfig.kt:134-143`

`properties` 和 `values` 的过滤不在 `configs` 赋值时立即执行，而是通过 `membersType == UNSET` 惰性触发（`CwtConfigResolverManager.kt:125-138`）：

- 若 `configs` 全为属性规则 → `membersType = PROPERTY`，`getProperties` 直接 `cast()`，无需分配新列表
- 若 `configs` 全为值规则 → 类似处理
- 混合情况才进行 `filterIsInstance`

这避免了绝大多数情况下（规则块通常是纯属性或纯值）的不必要列表分配。

### 3.5 CwtPropertyPointer 复合指针

**位置**：`CwtPropertyPointer.kt`、`CwtPropertyConfig.kt:153`

自定义的 `CwtPropertyPointer` 同时持有对 `CwtProperty` 和其属性值 `CwtValue` 的智能指针（`propertyValuePointer`）。这样在 `CwtValueConfig.valueConfig` 懒加载时（`CwtPropertyConfig.kt:219`），可以直接从 `resolvedPointer.propertyValuePointer` 取用，无需重新创建指针：

```kotlin
val valuePointer = when {
    resolvedPointer is CwtPropertyPointer -> resolvedPointer.propertyValuePointer
    else -> resolvedPointer.element?.propertyValue?.createPointer()
} ?: return null
```

### 3.6 DefaultProject 使用空指针

**位置**：`CwtPropertyConfig.kt:153`、`CwtValueConfig.kt:112`

对 `default project` 的 config group，`pointer` 直接使用 `emptyPointer()`（`PlatformExtensions.kt:669`，返回全局单例），不创建 `SmartPsiElementPointer`，避免 SmartPointerManager 的资源占用。

### 3.7 valueConfig 的懒加载 + 哨兵值

**位置**：`CwtPropertyConfig.kt:208-223`

`CwtPropertyConfigBase._valueConfig` 使用 `EMPTY_OBJECT` 作为哨兵值区分"未初始化"与"值为 null"，配合 `@Volatile` 和 `@Synchronized` 保证多线程安全的单次计算。

### 3.8 CwtOptionConfig / CwtOptionValueConfig 的强实例缓存

**位置**：`CwtOptionConfig.kt:69,116-122`、`CwtOptionValueConfig.kt:50,61-68`

对不含嵌套子选项的简单选项规则，通过 `CacheBuilder` 的复合 key 缓存（`"${valueType.ordinal}#${separatorType.ordinal}#${key}#${value}"`）实现相同内容的全局实例复用。由于选项注释中频繁出现重复的常见选项（如 `## cardinality = 0..1`、`## scope = country`），这个缓存能显著减少重复对象分配。

### 3.9 @InlinedDelegateFields 字节码注入

**位置**：`InlinedDelegateFieldCodeInjectors.kt:10-13`、`InlinedDelegateFieldCodeInjectorSupport.kt`

这是最具创意的优化。Kotlin 属性委托（`var prop by SomeKey`）会在每个实例中生成一个 `prop$delegate` 实例字段，持有对委托对象（`RegistedKey` 实例）的引用（4 字节）。

`CwtOptionDataHolderBase` 定义了约 30 个委托属性（对应 30 个已知选项字段），理论上会在每个实例中产生 ~30 × 4 = **120 字节**的委托引用字段。

通过 Javassist 字节码注入（`@InlinedDelegateFields`），在类加载时将所有 `$delegate` 字段的读取替换为对静态常量的直接引用，并移除实例字段本身。由于 `RegistedKey` 实例是静态对象（`Keys.cardinality` 等），替换为静态访问是等价且安全的。

最终效果：每个继承自 `CwtOptionDataHolderBase` 的规则实例节省约 **120 字节**的委托引用字段。

### 3.10 按是否有子规则分拆实现类

**位置**：`CwtPropertyConfig.kt:272-333`、`CwtValueConfig.kt:216-275`

`CwtPropertyConfigImpl`（叶节点）不持有 `configs` 和 `membersType` 两个字段，而是通过覆写 `configs` getter 返回条件性的结果：

```kotlin
// CwtPropertyConfigImpl
override val configs: List<CwtMemberConfig<*>>? 
    get() = if (valueTypeId == blockValueTypeId) emptyList() else null
```

相比统一持有 `configs: List?` 字段，叶节点每实例节省 8 字节（1 个列表引用 + 1 个枚举字段 `membersType`）。考虑到 CWT 规则树中叶节点数量远多于块节点，这是一个有价值的分拆。

## 四、进一步优化空间分析

经过以上分析，可以识别若干尚有讨论余地的方向。总体而言，收益均为**边际性**，需要结合实际 heap profiling 数据判断是否值得处理。

### 4.1 _valueConfig 字段的按需存储

**位置**：`CwtPropertyConfig.kt:208`

```kotlin
@Volatile private var _valueConfig: Any? = EMPTY_OBJECT
```

这个字段存在于**每一个** `CwtPropertyConfigBase` 实例中（包括叶节点和块节点），每实例占用 4 字节。然而：

- 对于**块节点**，`valueConfig` 指向其 value 侧的 `CwtValueConfig`，访问频率较高（用于获取子规则的表达式类型等）
- 对于**叶节点**，`valueConfig` 也是合理的访问目标（提供 `valueExpression`）
- 但对于**委托规则**（`CwtPropertyConfigDelegate` 等），`valueConfig` 的访问更多通过被委托对象间接触发，直接访问频率较低

若将 `_valueConfig` 的缓存迁移至 `UserDataHolder`（通过 `CwtMemberConfig.Keys` 注册一个专用 key），则每个不访问 `valueConfig` 的实例节省 4 字节，代价是首次访问多一次 map 查找。

**评估**：收益有限（4 字节/实例），且 `UserDataHolder` 内部 CHM 的首次 `put` 本身有内存成本。除非有明确的 profiling 数据证明 `valueConfig` 访问率极低，否则不建议优化。

### 4.2 membersType 字段的 Byte 化

**位置**：`CwtPropertyConfig.kt:299`、`CwtValueConfig.kt:241`

```kotlin
@Volatile private var membersType: CwtMembersType = CwtMembersType.MIXED
```

`CwtMembersType` 只有 5 个值（`UNSET/NONE/MIXED/PROPERTY/VALUE`），可以仿照 `CwtType` 用 `Byte` 存储以节省对象引用。但这个字段仅存在于"带子规则"的变体类（`ImplWithConfigs` 和 `DelegateWithConfigs`）中，而这些变体是少数，整体数量远少于叶节点。

**评估**：影响范围小，收益微弱，不建议实施。

### 4.3 CwtOptionConfig 缓存 key 的字符串分配

**位置**：`CwtOptionConfig.kt:116`、`CwtOptionValueConfig.kt:63`

```kotlin
val cacheKey = "${valueType.ordinal}#${separatorType.ordinal}#${key}#${value}"
```

每次 `create` 调用（选项规则不含嵌套子项时）都会拼接一个临时字符串作为缓存 key，产生短生命周期的 `String` 分配。

可替换方案：
- 使用嵌套 map（`ByteArray → Map<String, Map<String, CwtOptionConfig>>`）
- 使用 data class 作为 key（需实现 `equals`/`hashCode`，可以避免字符串分配）

**评估**：这是 config 加载期的一次性开销，插件启动后不再调用，对运行时内存无持续影响。不建议优化。

### 4.4 UserDataHolderBase 内部 CHM 的初始化行为确认

**位置**：`CwtOptionDataHolderBase.kt`（继承 `UserDataHolderBase`）

每个 `CwtPropertyConfigBase` / `CwtValueConfigBase` 实例都继承 `UserDataHolderBase`，后者内部使用 `ConcurrentHashMap`（或 IntelliJ 的定制实现）存储 key-value 数据。

若 `UserDataHolderBase.openMap()` 是**惰性初始化**（首次 `put` 时才分配），则对于从未写入任何 UserData 的规则实例（纯叶节点，无 `singleAliasConfig`、`aliasConfig` 等附加数据），CHM 字段为 `null`，内存开销可接受。

若 CHM 是**急切初始化**（构造时即分配），则每个规则实例会多出一个 CHM 对象头（16 字节）+ 初始 table（即使是最小容量也有一定开销）。

**建议**：通过实际 heap dump 或 JOL 分析确认 IntelliJ 版本的 `UserDataHolderBase` 初始化行为，若为急切初始化则此处有优化空间（例如使用 lazy 包装或自定义简单 map 替代）。

### 4.5 @Volatile 字段的访问模式评估

**位置**：`CwtPropertyConfig.kt:205,208`、`CwtValueConfig.kt:174,240`

```kotlin
@Volatile override var parentConfig: CwtMemberConfig<*>? = null
@Volatile override var configs: List<CwtMemberConfig<*>> = emptyList()
```

`@Volatile` 字段在 JVM 内存模型中不额外占用空间，但每次读写都会产生 store/load 内存屏障，禁止重排序，有一定的 CPU 开销（非内存开销）。

对于 `parentConfig`，它在 `postProcess`/`postOptimize` 阶段写入一次，此后在语言功能（补全、解析等）的高频读路径中只读。若能保证初始化完成后只读（即 config group 加载完成后所有规则对象进入"只读阶段"），则可以考虑将写入集中在初始化阶段并提供"发布语义"（如通过 `@Volatile` 的 config group 引用传递，从而省略各字段的 `@Volatile`）。

**评估**：这涉及对 config 生命周期的系统性理解，改动影响面较大，风险高于收益，不建议优化。

## 五、总体评价

| 维度 | 评价 |
|---|---|
| **接口设计** | 清晰，职责划分合理。`CwtDetachedConfig`、`CwtDelegatedConfig`、`CwtMemberContainerConfig` 的边界处理得当，`UserDataHolder` 的继承设计为扩展属性提供了统一的存储机制 |
| **实现类分层** | 按"是否有子规则"、"是否是委托"分拆实现类，严格遵循最小字段原则，设计到位 |
| **内存优化覆盖面** | 相当全面：enum → Byte、String interning、ImmutableList、SmartPointer 复用、EmptyPointer、`@InlinedDelegateFields`、实例缓存、延迟计算等均已落地 |
| **字节码注入** | `InlinedDelegateField` 机制是少见但有效的手段，对 `CwtOptionDataHolderBase` 每实例节省约 120 字节，是最显著的单项优化 |
| **进一步优化空间** | 有限，主要集中在 `_valueConfig` 按需存储（收益 4 字节/实例，代价是访问性能降低）和 `UserDataHolderBase` 初始化行为的确认（需 heap profiling 验证）；其余方向收益微弱或改动风险较高 |

**结论**：这套设计在同类 IDE 插件中属于内存意识相当强的实现，已经做了超出常规 Kotlin 代码的精细优化。当前的设计方案在工程可维护性与内存效率之间取得了良好的平衡，主要剩余优化空间需要实际 heap profiling 数据来驱动，不建议在无量化依据的情况下盲目调整。
