#### 重构 `CwtDataTypes` `CwtConfigTypes` - 改为偏好使用 lambda 式构建器

- 版本：v2.1.8
- 日期：2026-05-04

说明：
- 改为偏好使用 lambda 式构建器，而非多行的链式构建器。
- `.suffixAware()` `.reference()` 这样的“重要信息”需要紧随 `builder()` 之后，其余细节信息放到 `builder {...}` 中。
- 如果没有细节信息，直接使用单行的链式构建器即可。

目的：
- 为了适用代码折叠，从而隐藏细节，方便查看。

相关代码位置：
- `icu.windea.pls.config.CwtConfigTypes`
- `icu.windea.pls.config.CwtDataTypes`

#### 重构作用域和作用域方法 - 改为偏好使用成员方法作为作用域方法，而非带上下文参数的扩展方法

- 版本：v2.1.8
- 日期：2026-05-08

说明：
- 这里的作用域指的是某种接口或类，对应某种限定的API范围，某些情况下还带有自己的上下文。`SequenceScope` 和 `CoroutineScope` 都是典型的作用域。
- 区别于普通的工具方法和扩展方法，作用域方法只能在对应的作用域中使用。例如，将作用域对象作为 `this`，或者上下文参数 `context(scope: Scope)`。
- 改为偏好使用成员方法作为作用域方法，而非带上下文参数的扩展方法。除非不存在一个明确的可控制的作用域（类/接口）。

目的：
- 即使上下文中不存在符合条件的作用域对象，IDE 仍会补全作为扩展方法的那些作用域方法。
- 改为使用成员方法的话，可以避免这种非理想的行为，并且允许隐藏实现细节。

相关代码位置：
- `icu.windea.pls.config.select.CwtConfigSelectScope`
- `icu.windea.pls.lang.select.ParadoxPsiSelectScope`

#### 重构 path/expression/complexExpression/config/configExpression - 更改解析器与解析方法的代码风格

- 版本：v2.1.10
- 日期：2026-06-14

说明：
- 将对外的解析方法改为模型接口上的静态方法，委托给解析器类（可能是单例对象，可能是私有的）的对应方法。
- 解析器类通常是同文件中的私有类，少数情况下也可以是不同文件中的公开类。
- 偏好上，将同文件中的私有实现方法放到公开接口方法之后，并且使用 `region...endregion` 注释包围。
- 偏好上，将文档注释放到接口代码上，实现代码（以及公开的解析器类的代码）通常不需要文档注释。

目的：
- 为了统一代码风格。

示例（不需要公开解析类的情况）：

```kotlin
interface Expression {
    val value: String
    
    companion object {
        @JvmStatic
        fun resolve(text: String): Expression = ExpressionResolver.resolve(text)
    }
}

// region Implementations

private object ExpressionResolver {
    fun resolve(text: String): Expression {
        return ExpressionImpl(text)
    }
}

private data class ExpressionImpl(override val value: String)

// endregion
```

#### 关于上下文对象（context object）

- 版本：v2.1.10
- 日期：2026-06-20

说明：
- 上下文对象应当被声明为数据类，其属性应尽可能不可变，可以有计算属性和一些工具方法。
- 上下文对象的切换是通过复制当前上下文，并按需更改特定的属性来达成的。
- 切换上下文时，不一定需要直接传递复制后的上下文对象，或是声明并传递不同名的局部变量。
- 一般来说，上下文的切换可以随编程语言（Kotlin）的作用域的切换而按需切换，直接声明同名的局部变量并覆盖（如在循环中）。

#### 支持位置感知的数据类型

- 版本：v2.2.0
- 日期：2026-06-30

相关问题：
- https://github.com/DragonKnightOfBreeze/Paradox-Language-Support/issues/340

说明：
- 支持 *位置感知的数据类型*，适用于定义和本地化的匹配（如 `localisation|$_name,$_desc`），并替换掉现有的 *后缀感知的数据类型* （如 `localisation|_name,_desc`）。
- 例如，如果脚本表达式 `test` 匹配的数据表达式是 `localisation|$_name,$_desc`，则仅当同时存在本地化 `test_name` 和 `test_desc` 时才认为完全匹配。 
- 对于匹配这类数据类型的脚本表达式，目前暂不提供对应的代码补全和用法查询。
- 对于匹配这类数据类型的脚本表达式，认为进行引用解析时无法直接解析为对应的定义/本地化化，而是将之视为一种“相关项”。
- 对于匹配这类数据类型的脚本表达式，考虑提供这类“相关项”的导航动作、装订线图标（lineMarkerProvider）和自动重命名（automaticRenamer）。

新增数据类型：
- `DefinitionWithLocations` - 替换 `SuffixAwareDefinition` - 示例：`<job>|$_name,$_desc`
- `LocalisationWithLocations` - 替换 `SuffixAwareLocalisation` - 示例：`localisation|$_name,$_desc`
- `SyncedLocalisationWithLocations` - 替换 `SuffixAwareSyncedLocalisation` - 示例：`localisation_synced|$.name,$.desc`

#### 增强对定值（define）的支持

- 版本：v2.1.8
- 日期：2026-05-04

说明：
- 定值（define）分为定值的命名空间和变量，它们固定声明于游戏目录的 `common/defines` 子目录中的带 `.txt` 扩展名的脚本文件中。
- 引入对应的定值规则（defineConfig），从而为定值的命名空间和变量提供快速文档文本和规则上下文。
- 定值是在（游戏目录中的）脚本文件中声明的。普通导航会跳转到覆盖项，相关规则导航才会跳转到对应的定值规则。
- 理论上，定值是取决于具体的游戏的，并且会在游戏目录中声明并列出所有可用项。
- 但是，插件目前不通过定值规则检查定值的命名空间和变量的名字的合法性（仍然检查其声明结构的合法性，如同定义和定义注入一样）。
- 插件会强制忽略名为 `define` 或 `defines` 的类型规则和声明规则（参见：`icu.windea.pls.ep.config.config.CwtForcedConfigFilterProvider`）。
